# Commit Message Writer

## Overview

Produce accurate Git commit messages by inspecting the actual repository changes, matching the repository's existing commit style, writing the message content in Chinese by default, and keeping each commit focused on one understandable purpose.

## Workflow

1. Inspect the repository before writing the message.
   - Run `git status --short`.
   - If staged changes exist, treat `git diff --staged` as the commit source of truth.
   - If nothing is staged, inspect `git diff` and untracked file names. Do not assume untracked file contents without reading them when they matter.
   - Check recent style with `git log -5 --pretty=format:%s` when available.

2. Decide whether the changes form one coherent commit.
   - If the diff mixes unrelated work, recommend splitting it into separate commits and provide one message per logical group.
   - When the user says there are too many changes or they do not know how to split them, default to a step-by-step commit plan instead of a single message.
   - If the user explicitly wants one commit, write one message that honestly summarizes the combined change.

3. Split large changes by intent.
   - Group changes by user-visible feature, bug fix, refactor, documentation, tests, build/tooling, or cleanup.
   - Prefer commits that can be explained in one sentence and reviewed independently.
   - Keep prerequisite changes before dependents: build/tooling first, shared helpers before callers, implementation before tests only when the repository's history uses that order.
   - Do not split purely by file count. A single coherent feature may touch several files, and one file may contain unrelated hunks.
   - Separate formatting-only changes from behavior changes when practical.
   - Separate docs/tests from implementation when they are substantial or when doing so makes review clearer; keep them together when they directly document or verify one small change.

4. Build a commit plan before committing.
   - For each proposed commit, list: purpose, files or hunks to stage, commit message, and why it is separate.
   - If whole files map cleanly to one purpose, stage by file path.
   - If a file contains unrelated hunks, mention that partial staging is needed. Use `git add -p` carefully only when the user asked Codex to perform the commits; otherwise explain which hunks belong together.
   - If the working tree is too tangled to split safely, say so and recommend a smaller manual cleanup step before committing.
   - Avoid staging unrelated untracked files unless their purpose is clear from the request and inspected contents.

5. Match the repository's style.
   - Prefer the style already present in recent commits.
   - If no clear style exists, default to Conventional Commits for code repositories.
   - Use plain imperative subjects for repositories that do not use Conventional Commits.
   - Keep Conventional Commit `type` values in English, such as `feat`, `fix`, and `docs`.
   - Write the subject summary and body in Chinese by default, except for proper nouns, file names, commands, API names, symbols, and established technical terms that should stay in their original form.

6. Draft each message.
   - Keep the subject specific, imperative, and usually under 72 characters.
   - Do not end the subject with a period.
   - Mention the affected area as a scope when it is obvious.
   - Add a body only when it explains important motivation, risk, migration notes, or test coverage.
   - Never add "generated by AI", co-author trailers, or unrelated metadata unless the user asks.

7. If the user asked Codex to commit.
   - Use the drafted message with `git commit`.
   - Do not push unless the user explicitly asked to push.
   - If the index is empty, explain that there are no staged changes. Stage files only when the user's request clearly includes staging those changes or committing the working tree.
   - For multiple commits, stage and commit one logical group at a time, then re-check `git status --short` before the next commit.
   - After committing, report each commit hash and the exact message used.

## Default Format

When Conventional Commits are appropriate:

```text
type(scope): 简洁的中文祈使句摘要

可选正文，用中文说明原因、重要行为变化或验证方式。
```

Examples:

```text
feat(kernel): 添加串口启动日志

docs(tooling): 记录 QEMU 调试流程

build(make): 增加内核链接脚本配置
```

Common types:

- `feat`: user-visible feature or capability
- `fix`: bug fix
- `docs`: documentation-only change
- `test`: test-only change
- `refactor`: internal restructuring without intended behavior change
- `chore`: maintenance that does not fit another type
- `build`: build system or dependency change
- `ci`: CI configuration or workflow change
- `perf`: performance improvement
- `style`: formatting-only change

## Output Guidelines

When only asked to write one message, return the recommended message first in a fenced `text` block. Add brief alternatives only if they are materially useful.

When asked to split changes, return a concise commit plan:

```text
1. type(scope): 第一条聚焦的中文提交信息
   Stage: path/to/file, selected hunks in path/to/other-file
   Why: 用中文简要说明为什么这是一个独立提交

2. type(scope): 第二条聚焦的中文提交信息
   Stage: path/to/file
   Why: 用中文简要说明为什么它应该拆开
```

When reviewing a proposed commit message, check whether it is accurate, specific, in the repository's style, and appropriately scoped. Provide a corrected version if needed.

When there are no meaningful changes, say so instead of inventing a message.
